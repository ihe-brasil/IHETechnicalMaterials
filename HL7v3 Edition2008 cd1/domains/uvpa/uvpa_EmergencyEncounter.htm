Introduction

This topic covers messages about Emergency Encounters in the Event mood. In an emergency encounter a patient is admitted for immediate evaluation and therapy provided until the patient can be referred to his or per personal practitioner for further care. In HL7 version 2.x terms, an emergency encounter is a visit with a Patient Class (PV1-2) of "emergency". Because emergency encounters are not planned, no appointment or associated preadmission encounter is included in the information models.

Scope

Event Notifications

This release supports five event notifications. The message content is defined at:

  • Begin Emergency Encounter (see Emergency Encounter Event Activate)

  • Revise Emergency Encounter (see Emergency Encounter Event Revise)

  • End Emergency Encounter (see Emergency Encounter Event Complete)

  • Abort Emergency Encounter (see Emergency Encounter Event Abort)

  • Nullify Emergency Encounter (see Emergency Encounter Event Nullify)

Scope Restrictions

The Patient Administration Technical Committee chose to limit scope in this first release by:

  • Limiting storyboards to the simplest cases

  • Excluding order/request messages

  • Excluding notification messages for state transitions "reactivate" and to/from "suspended" (see the Act State Diagram for Patient Administration Messages below; the unsupported states and transitions are grayed out in the diagram)



Act State Diagram for Patient Encounter Messages

The Patient Administration Technical Committee invites implementers with additional requirements to submit content proposals for future releases of the standard.

 8.1.1.1Emergency Encounter Basic Flow (PRPA_SN403001UV02)

Emergency Check-In

Adam Everyman arrived at Good Health Hospital emergency room via ambulance. Mr. Everyman was in respiratory distress and had an accelerated heart beat. The physician on duty, Dr. Eric Emergency, decided Mr. Everyman should be treated at this time. Mr. Everyman was checked-in for an ER visit. Christopher Clerk, the emergency room clerk, found Mr. Everyman's record in the GHH Patient Registry and created the emergency check-in [Trigger Event Begin Emergency Encounter].

Revise Active Emergency Encounter

The GHH ER clerk, Christopher, reviewed the contact information in Adam Everyman's patient record with him. Mr. Everyman stated that he needed to change his emergency contact information. Mr. Everyman's daughter was out of town so Mr. Everyman informed Christopher that he wanted to put his son, James Everyman, down as the emergency contact. He provided Christopher with James' phone number and address [Trigger Event Revise Emergency Encounter].

The GHH ER specialist, Dr. Eric Emergency decided that after a nebulizer treatment Mr. Everyman was stable and was ready to be checked-out. Dr. Emergency noted that Mr. Everyman needed to schedule a follow-up visit with Dr. Puffer, pulmonologist.

Check-Out

The GHH ER clerk completed the check-out information for Mr. Everyman and checked him out of the GHH Emergency Room [Trigger Event End Emergency Encounter].

Post Check-Out

Upon post check-out review, the Medical Records director discovered a diagnosis coding error. She corrected the coding to accurately reflect the diagnosis [Trigger Event Revise Emergency Encounter].

8.3.1Begin Emergency Encounter  (PRPA_TE403001UV02)
Description
Structured Name:  Emergency Encounter Event Activate Notification
Type:  State-transition based
State Transition:  EncounterEvent (PRPA_RM403001UV02)

The Begin Emergency Encounter trigger event signals that a new emergency encounter started. This is sometimes called an emergency admission.


V2 Reference

The Begin Emergency Encounter trigger event is aligned with two HL7 2.4 events, the ADT^A04 - register a patient event and the ADT^A01 - admit/visit notification. In both cases the patient class in the PV1-2 is emergency.

8.3.2Revise Emergency Encounter  (PRPA_TE403002UV02)
Description
Structured Name:  Emergency Encounter Event Revise Notification
Type:  State-transition based
State Transition:  EncounterEvent (PRPA_RM403002UV02)

The Revise Emergency Encounter trigger event signals that information about an emergency encounter has changed.


V2 Reference

The Revise Emergency Encounter trigger event is most closely aligned with the HL7 2.4 ADT^A08 - update patient information event where the patient class in the PV1-2 is emergency. However, in v3 this trigger event only signals a change to encounter information, changes to a patient's demographic information are signaled by a Revise Patient Information trigger event.

8.3.3Abort Emergency Encounter  (PRPA_TE403004UV02)
Description
Structured Name:  Emergency Encounter Event Abort Notification
Type:  State-transition based
State Transition:  EncounterEvent (PRPA_RM403004UV02)

The Abort Emergency Encounter trigger event signals that an emergency encounter was aborted prior to completion. Either the practitioner or the patient ended the encounter prematurely.


V2 Reference

The Abort Emergency Encounter trigger event is most closely aligned with the HL7 2.4 ADT^A03 - discharge/end visit event where the patient class in the PV1-2 is emergency and the discharge disposition in the PV1-36 is "Left against medical advice or discontinued care."

8.3.4End Emergency Encounter  (PRPA_TE403003UV02)
Description
Structured Name:  Emergency Encounter Event Complete Notification
Type:  State-transition based
State Transition:  EncounterEvent (PRPA_RM403003UV02)

The End Emergency Encounter trigger event signals that an emergency encounter ended normally. This is variously called end visit or emergency encounter check-out.


V2 Reference

The End Emergency Encounter trigger event is most closely aligned with the HL7 2.4 ADT^A03 - discharge/end visit event where the patient class in the PV1-2 is emergency.

8.3.5Nullify Emergency Encounter  (PRPA_TE403999UV02)
Description
Structured Name:  Emergency Encounter Event Nullify Notification
Type:  State-transition based
State Transition:  EncounterEvent (PRPA_RM403005UV02)

The Nullify Emergency Encounter trigger event signals that a previously-reported Begin Emergency Encounter notification was sent in error and has been nullified.


V2 Reference

The Nullify Emergency Encounter trigger event is most closely aligned with the HL7 2.4 ADT^A11 - cancel admit/visit notification event where the patient class in the PV1-2 is emergency.

8.4.1Emergency Encounter Event Activate  (PRPA_RM403001UV02)

Diagram


Click thumbnail above to open larger graphic in a new window
Description
Parent:   Patient Administration (PRPA_DM000000UV)

The Emergency Encounter Event Active R-MIM defines the message used to report that a new emergency encounter has started.

Encounter Attributes

The focal class (and entry point) of the Emergency Encounter Event Activate R-MIM is the EncounterEvent class. The attributes of encounter are:

  • id - one or more unique identifiers for this encounter (for example, an "internal" id used only by computer systems and an "external" id for display to users)

  • code = EMER, says that this is an emergency encounter

  • statusCode = "active"

  • effectiveTime - a starting time representing the administrative onset of the encounter (e.g. admission, registration, patient arrival) and, optionally, an ending time representing the anticipated date-time of the patient's departure (e.g. discharge). All messages, except the "nullify" message, require at least a starting time.

  • activityTime - the total time interval for an encounter including preparation before the patient arrives and clean-up actions after the patient departs. This would probably be null in an encounter activate message.

  • priorityCode - the encounter admission urgency.

  • confidentialityCode - a code that controls the disclosure of information about this patient encounter.

  • reasonCode - gives an explanation for the encounter, other than the medical reason for the encounter in the pertinentInformation1 (diagnoses) described below. Examples of values here are "Medical Necessity", "Patient's Request" and "Dependency".

  • admissionReferralSourceCode - a code used to categorize the type of place or organization responsible for the patient immediately prior to their admission. In the US realm this references the UB92 FL 20 - Source of Admission value set.

  • lengthOfStayQuantity - expresses the expected quantity of time for the encounter.

  • dischargeDispositionCode - A code depicting the expected disposition of the patient at the time of discharge. In the US realm this references the UB92 FL 22 - Patient Status value set.

  • acuityLevelCode - a code depicting the acuity (complexity of patient care, resource intensiveness of the patient care) of a patient's medical condition upon arrival.

  • specialCourtesiesCode - a code identifying special courtesies extended to the patient (e.g., no courtesies, extended courtesies, professional courtesy, VIP courtesies).

  • specialArrangementCode - a code indicating the type of special arrangements provided for a patient encounter (e.g., wheelchair, stretcher, interpreter, attendant, seeing eye dog). This is not related to the AccommodationEvent.

Links to Other Encounters

Encounters can relate to other encounters in three ways:

  • sequelTo: this can link the encounter to a previous encounter for the same patient. Examples include: (1) a patient in an emergency encounter is admitted as an emergency after an evaluation of the seriousness of the patient's condition, (2) an emergency is discharged from an emergency encounter directly to an emergency encounter as part of the same episode of care, (3) a patient is re-admitted to a healthcare facility. Context conduction ensures that both encounters have the same subject (patient).

  • componentOf: this can link the encounter to one or more sub-encounters performed within the main encounter. Examples include: (1) an outpatient visit includes clinical encounters with radiology and clinical specialists, (2) an emergency room visit includes encounters for two non-related issues (blurry vision in the eye and a sprained knee, causing the patient to be sent to an eye specialist and an orthopedist). Context conduction ensures that all encounters have the same subject (patient)

Links to Other Acts

Encounters can link to other acts:

  • AccommodationEvent: the patient's assigned location during an encounter, ServiceDeliveryLocation, can be linked to one or more AccommodationEvent acts via a locationOf participation. A patient may request or need a specific accommodation such as ward, private, or semi-private accommodations for (a part of) the encounter. The AccommodationEvent.reasonCode tells if the accommodation is a medical necessity or the result of a patient request.

  • PatientTransportation: the patient's arrival can be linked to the encounter via an arrivedBy act relationship. The patient's mode of arrival can be sent as PatientTransportation.text, the admitted-from location in a R_LocationLocatedEntity CMET, and the time patient arrived at the encounter site as the end of thePatientTransportation.effectiveTime range. Both the performer (person who transported the patient) and an escort can be sent in R_AssignedPerson [contact] CMETs. Context conduction ensures that the encounter and the transportation have the same subject (patient).

  • A_Account: the encounter can be linked to one or more patient accounts. Each account is sent in a separate A_Account CMET.

  • A_ObservationDx: the encounter can be linked to one or more diagnoses. Each diagnosis is sent in a separate A_ObservationDx CMET. The encounter "activate" interaction would send an admission diagnosis (Observation.code of "ADMX" in the A_ObservationDx CMET). Multiple admission diagnoses can be rank ordered using the pertinentInformation1.priorityNumber.

  • A_Consent: the encounter can be linked to one or more consents granted by the patient or patient's representative. Each consent is sent in a separate A_Consent CMET. Note, this is currently a placeholder CMET until the Medical Records TC completes work on medico-legal document modeling.

  • ValuablesLocation: the encounter can be linked to one or more reports of patient valuables stored during the encounter. Each report is sent as a free text observation of the valuables and where they are stored during the encounter. If no valuables were stored for the patient then an ValuablesLocation can be sent with a negationInd of "true".

Direct Participations in the Encounter

A number of entities play roles that participate directly in the encounter:

  • admitter: the healthcare practitioner that required/authorized the encounter, usually in the case of an emergency encounter. Note: the admitter here must be the same as the author reported in the HL7 Trigger Event Control Act for the admission.

  • responsibleParty: the healthcare provider organization that holds clinical responsibility for care of the patient during the encounter. The encounter "activate" interaction activates the initial responsibleParty participation.

  • location: the patient's assigned location during the encounter. The encounter "activate" interaction activates the initial location participation. ServiceDeliveryLocation is a role played by a place at which services may be provided. A service delivery location may be either an incidental service delivery location (a place at which services may be provided without prior designation or authorization) or a dedicated service delivery location (a place that is intended to house the provision of services). Dedicated service delivery locations can be further characterized as either clinical (DedicatedClinicalLocationRoleType) or non-clinical (DedicatedNonClinicalLocationRoleType).

  • subject: the patient who is the subject of the encounter.

  • referrer: the healthcare practitioner who requested the encounter to take place.

  • consultant: an advisor participating in the encounter by performing evaluations and making recommendations.

  • attender: the healthcare practitioner who is primarily responsible for the patient during a period in time in the given encounter. The encounter "activate" interaction activates the initial attender participation.

  • notificationContact: the patient's designated Emergency Contact for this encounter.

8.4.2Emergency Encounter Event Revise  (PRPA_RM403002UV02)

Diagram


Click thumbnail above to open larger graphic in a new window
Description
Parent:   Patient Administration (PRPA_DM000000UV)

The Emergency Encounter Event Revise R-MIM defines the message used to report that information about an emergency encounter has changed. The emergency encounter can be either active or completed.

Differences

Emergency Encounter Event Revise R-MIM differs from the Emergency Encounter Event Activate R-MIM in the following ways:

  • EncounterEvent does not include the statusCode attribute because state changes in EncounterEvent are conveyed by the trigger event identified in the Trigger Event Control Act.

  • responsibleParty - the encounter revise message includes the current responsibleParty but should not be used to change the active responsibleParty. This is done with a message derived from the Encounter Organization Change R-MIM.

  • discharger - the encounter revise message includes an optional discharger participation in EncounterEvent. The discharger is the healthcare practitioner that required/authorized the ending of an encounter.

  • location - the encounter revise message includes the current location but should not be used to change the active location. This is done with a message derived from the Encounter Location Change R-MIM.

  • The single arrivedBy act relationship from EncounterEvent to PatientTransportation is replaced with up to two transportedBy act relationships permitting both patient arrival and patient departure transportation events.

  • attender - the encounter revise message includes the current attender but should not be used to change the active attender. This is done with a message derived from the Attending Practitioner Change R-MIM.

8.4.3Emergency Encounter Event Abort  (PRPA_RM403004UV02)

Diagram


Click thumbnail above to open larger graphic in a new window
Description
Parent:   Patient Administration (PRPA_DM000000UV)

The Emergency Encounter Event Abort R-MIM defines the message used to report that an emergency encounter was aborted prior to normal completion. The composite message aborts the encounter, completes the managed participations in the encounter, and conveys the reason the encounter was aborted.

ControlActProcess

The reason the encounter was aborted is conveyed in the Trigger Event Control Act portion of the HL7 composite message. The reasonCode attribute of the ControlActProcess conveys a simple, coded reason. More comprehensive information about why the encounter was aborted may be conveyed in a A_DetectedIssue CMET associated with the ControlActProcess via a reasonOf act relationship.

EncounterEvent

The focal class (and entry point) of the Emergency Encounter Event Abort R-MIM is the EncounterEvent class. The attributes of encounter are:

  • id - one or more unique identifiers for this encounter (for example, an "internal" id used only by computer systems and an "external" id for display to users)

  • code = EMER, says that this is an emergency encounter

  • statusCode = "aborted"

  • effectiveTime - the time interval starting with the administrative onset of the encounter (e.g. admission, registration, patient arrival) and ending with the patient's departure (e.g. discharge).

  • activityTime - the total time interval for an encounter including preparation before the patient arrives and clean-up actions after the patient departs.

  • reasonCode - gives an explanation for the encounter, other than the medical reason for the encounter. Examples of values here are "Medical Necessity", "Patient's Request" and "Dependency".

  • lengthOfStayQuantity - expresses the actual quantity of time for the encounter.

  • dischargeDispositionCode - A code depicting the disposition of the patient at the time of discharge (e.g., discharged to home, expired, against medical advice, etc.). In the US realm this references the UB92 FL 22 - Patient Status value set.

EncounterEvent Links to Other Acts

Encounters can link to other acts:

  • AccommodationEvent: the patient's assigned location during an encounter, ServiceDeliveryLocation, can be linked to one or more AccommodationEvent acts via a locationOf participation. Generally an emergency encounter would not include an AccommodationEvent but it is included to support use cases such as emergency surgery and observation patients.

  • PatientTransportation: the patient's departure can be linked to the encounter act via a departedBy act relationship. The patient's mode of departure can be sent as PatientTransportation.text, the discharged-to location in a R_LocationLocatedEntity CMET, and the time patient arrived (or is anticipated to arrive) at the discharged-to site as the end of thePatientTransportation.effectiveTime range. Both the performer (person who transported the patient) and an escort can be sent in R_AssignedPerson [contact] CMETs. Context conduction ensures that the encounter and the transportation have the same subject (patient).

Direct Participations in EncounterEvent

A number of entities play roles that participate directly in the encounter:

  • responsibleParty: the healthcare provider organization that holds clinical responsibility for care of the patient during the encounter. An encounter abort completes the current responsibleParty participation.

  • location: the patient's assigned location during the encounter. An encounter abort completes the current location participation.

  • subject: the patient who is the subject of the encounter.

  • attender: the healthcare practitioner who is primarily responsible for the patient during a period in time in the given encounter. An encounter abort completes the current attender participation.

  • notificationContact: the patient's designated Emergency Contact for this encounter in case they need to be contacted about the unexpected end of the encounter.

8.4.4Emergency Encounter Event Complete  (PRPA_RM403003UV02)

Diagram


Click thumbnail above to open larger graphic in a new window
Description
Parent:   Patient Administration (PRPA_DM000000UV)

The Emergency Encounter Event Complete R-MIM defines the message used to report that an emergency encounter has ended normally

Differences

Emergency Encounter Event Complete R-MIM differs from the Emergency Encounter Event Activate R-MIM in the following ways:

  • The EncounterEvent.statusCode is set to "completed".

  • The EncounterEvent.effectiveTime contains actual values for both start and end date-times instead of an anticipated end date-time.

  • The EncounterEvent.activityTime is more likely to contain a value.

  • The EncounterEvent.lengthOfStayQuantity, if valued, contains the actual, calculated quantity (the actual days quantity cannot be simply calculated from the admission and discharge dates because of possible leaves of absence) instead of the expected length of stay.

  • The EncounterEvent.dischargeDispositionCode, if valued, contains the actual discharge disposition instead of the expected discharge disposition.

  • The departedBy act relationship from EncounterEvent to PatientTransportation replaces the arrivedBy act relationship. The patient's mode of departure can be sent as PatientTransportation.text, the discharged-to location in a R_LocationLocatedEntity CMET, and the time the patient departed from the encounter site as the start of thePatientTransportation.effectiveTime range and time the patient arrived (or is expected to arrive) at the discharged-to location as the end of the time range. Both the performer (person who transported the patient) and an escort can be sent in R_AssignedPerson [contact] CMETs. Context conduction ensures that the encounter and the transportation have the same subject (patient).

  • There is an optional discharger participation in EncounterEvent. The discharger is the healthcare practitioner that required/authorized the ending of an encounter.
