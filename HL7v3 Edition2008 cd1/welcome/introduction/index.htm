Welcome

The HL7 Board of Directors is pleased to release the third Normative Edition of HL7 Version 3 Standards.  This Normative Edition contains all Version 3 Standards that are ANSI approved as of their publication date.  In circumstances where the normative specifications have a dependency upon material that is not yet normative, that additional not-yet-normative material is included in this edition. In the current release, this not-yet-normative material is limited to a handful of Common Message Element Types. 

In the event that a subsequent version of this edition is released it will not include any changes in the normative or processable content.  Version changes will be limited to alterations in the layout and supporting information.  However, this release does contain changes (in the form of later versions of specific artifacts) to content included in the previous Normative Edition Release, Edition 2006.

Normative Content

Ballot and ANSI status

With three exceptions, all of the specifications in this edition completed membership balloting as either a full standard, or as a Draft Standard for Trial Use (DSTU) and have been registered with the American National Standards Institute (ANSI).  Icons within each individual specification provide a confirmatory display of this status.  The exceptions to this statement are:
  • Vocabulary content – Much of the vocabulary table content in HL7 Version 3 is approved by a Harmonization process rather than by ballot.  Thus this content is shown with a "reference" icon.  Nevertheless, the vocabulary content is subject to negative votes whenever it is used in a balloted specification.  Thus, most of the vocabulary that controls the Reference Information Model was balloted when the RIM was approved as a standard.  Further, individual vocabulary constraints documented in the vocabulary have been considered when the artifacts within which those constraints were balloted. 
  • Selected Common Message Element Types (CMETs) – A few of the message designs in the normative material reference CMETs that have not completed balloting.  These required CMETs are also published in the CMET domain, but their ballot status icons (and their identifiers) distinguish them from the normative CMETs.
  • Informative Documents – These documents passed formal balloting as informative content. However, these documents are not considered normative and have not been registered with ANSI.

Artifact identifiers
(go to top)

Prior to their adoption as part of a normative specification, all artifacts carry a 13-character identifier, such as "COCT_MT987654".  These are the identifiers used in developing and publishing ballots.  With the release of the first Normative Edition in 2005, the identifiers for artifacts that are normative were extended with a four-character extension such as "COCT_MT987654UV02".   Interpretation of the additional characters is as follows:
  • Realm – The first two characters, "UV", indicate the realm.  In all cases in this edition the realm is "universal." (UV)
  • Release – The second two characters represent the version of the artifact.  This corresponds to normative release of the domain in which the artifact completed balloting.
Note that not-yet-normative artifacts (those still undergoing ballot) included in this edition do not include the full extension, although they will carry the first two-character, realm "UV" extension. This provides a second method, in addition to the icons for recognizing non-normative artifacts.

Artifact dependency
(go to top)

The HL7 Version 3 Methodology defines a set of relationships between artifacts that implies a dependency of one artifact on another.  For example, a Domain defines multiple Interactions, and an Interaction includes a message wrapper (Message Type), and the message wrapper design references a CMET.  Each of these defines a dependency – the complete definition of the Domain is dependent upon its complete set of Interactions; each of the Interactions is dependent upon the Message Type that defines the message wrapper; each of the wrapper Message Types may be dependent upon one or more CMETs; and so on.  These dependencies may be direct - Domain to Interaction in the example above - or indirect - Domain to CMET in the example above.

As part of the assembly of each Edition, all dependencies for each artifact in the Edition have been documented and validated.  (Validation is the verification that the artifact that satisfies a particular dependency is, indeed, also present in the edition.)  The results of this analysis are included in this package in three forms:
  • Domain dependency display - An HTML rendering of the complete dependencies for each domain is included as an Annex for that domain, and can be reached by a hyperlink from the domain table of contents.
  • Edition dependency tree – The complete dependency tree for the domain is included as an XML file.  Note that this shows the direct dependencies only.  In order to see the indirect dependencies, the file must be walked iteratively with X-path, as was done in creating the Interaction-message dependencies (below).  The element names in the tree are based on the identifier coding structure, such as element IN for an Interaction.  There are two exceptions to this rule:
    • Domain topic – A topic within a domain is identified with an element <BC/>, where 'BC' derives from base class.
    • Dependency – A dependency is shown as an element <dep kind=""> where the value of the "kind" attribute shows the type of element that satisfies the dependency.
  • Dependency error report – This report results from a transformation that validates all of the dependencies, and lists any detected omissions.  In this edition, this list is limited to a handful of errors in one of the non-normative domains.
  • Complete Interaction Dependencies – This report lists the complete set of "message type" (wrappers, payloads and CMETs) dependencies that exist for each interaction in each domain, and for the domain as a whole.  It aggregates the results by domain.  Within a domain, it lists each interaction, finds the interaction's "direct" dependencies, and then searches out the dependencies of those, until the full list is in hand.  It then sifts the unique set of these and sorts them as MT artifacts first and CT second, ordered by artifact ID within each group.  Finally, it lists the comprehensive list of all such dependencies within the domain.  These lists are useful in aggregating the complete set of schemas needed to support a particular interaction or domain.

Contents of CD Set
(go to top)

This Normative Edition is distributed as a two-CD set.  The first CD (which includes this file) is the Standards Publication disc and contains the complete documentation for the Edition in a form that supports HTML review of the contents.  The second CD, Processable Files and Tools, contains both a set of files (schemas, model design files, etc.) that will benefit an implementer and can be imported, edited and manipulated with software, and a set of HL7-sponsored tools that create and manipulate those files.  The contents of the two CDs can be merged.

Top Level Directory Structure of Both CDs

The root of these CD contains the following directories:
  • domains (Disc 1 only) – holds the complete normative expression (HTML, graphics, Excel) for each balloted domain. 
  • infrastructure (Disc 1 only) – holds the complete normative expression (HTML, graphics, Excel) for standards such as the RIM, data types, ITS, etc.
  • processable (Both discs) – contains files that hold computer-processable representations of the normative standards such as schemas, Visio designs, data bases, XML "MIF" and "hmd" files, etc.  (Disc 1 contains only the schemas.)
  • help (Disc 1 only) – includes content like the V3 guide and style guides.
  • welcome (Both discs) – holds this and related files.
  • support (Disc 1 only) – are a set of files (style sheets, graphics, etc.) that support the HTML representation of the standard.
  • source (Disc 2 only) – are a set of files - publication data bases, design repositories and RMIM Designer source files - that define the domain constant for the Edition.
  • toolfiles (Disc 2 only) – Installers and ZIP files holding the version of the HL7 Tools that were used to produce this Edition.

Domain Subdirectories (Disc 1 only)
(go to top)

Underneath the 'domains' subdirectory there is one subdirectory for each domain.  These are identified by the characters "uv" (for universal) followed by the domain's two-character id, yielding, for example "uvab" for Accounting and Billing.  Each of these domain directories contains the primary domain HTML content and has the following sub-directory structure:
  • editable – holds the Excel views of the domain's HMDs and the table-views in html ("*-NoEdit.htm"), for each HMD and Message Type in the domain.
    • images – the PNG files for the clickable RMIM design views
  • graphics – graphic material referenced from the domain's HTML renderings.

Infrastructure Subdirectories (Disc 1 only)
(go to top)

Underneath the 'infrastructure' directory, there is one directory for each of the infrastructure specifications.  These directories have further sub-directories, as required by each specification, but each includes a graphics sub-directory for the images referenced in the specifications HTML expression.

Processable Content Subdirectory (Disc 1)
(go to top)

In the 'processable' directory of Disc 1, this edition provides the set of schemas and examples needed to understand the specification.  The contents of this directory, by sub-directory, are:
  • coreschemas – This directory holds the schemas for data-types and vocabulary which are referenced (included) by all of the other schemas.  In order for the reference to work, the referring schema must be in a directory that is a 'sibling' of "coreschemas."
  • examples – Contains instance examples of messages where those have been defined.
  • multicacheschemas – This directory contains all of the other schemas – wrappers, CMETs and payloads – that are defined in the normative edition.
  • dependencytree – Contains the generated artifact dependency tree in XML along with an associated error report, if one is needed. The absence of an error report indicates that there are no significant dependency elements that are not specified.

Processable Content Subdirectory (Disc 2)
(go to top)

The 'processable' directory of Disc 2 contains the entire content of the 'processable' directory of Disc 1, but with the significant additions.  Of particular interest are the standard specifications expressed in HL7 Model Interchange Format (MIF).  There is a growing suite of tools distributed by HL7 and others that work with MIF files to create, manipulate and implement model artifacts.  Among the tools distributed here that use this format are the V3 Generator and RoseTree.  'processable' sub-directories from Disc 2 include:
  • mif – The complete set of designs represented in HL7 Model Interchange Format (MIF) Release 1.  (Schemas for this format are distributed as part of the V3 Generator tool.)
  • mif2 – The complete set of designs represented in HL7 Model Interchange Format (MIF) Release 2.1.  (Schemas for this format are distributed as part of the V3 Generator tool.)
  • mif-lite – The complete set of designs represented in HL7 Model Interchange Format (MIF) Release 2.1, but stripped of their descriptive documentation.  This form of MIF files will be distributed with a license that permits them to be used as configuration files for tools.

Source Subdirectory (Disc 2 only)
(go to top)

In the 'source' directory of Disc 2, this edition provides the complete set of source files used to build this edition, along with the complete set of source files needed to run the V3 Generator.  The contents of this directory, by sub-directory, are:
  • databases – Holds the publishing data bases (PubDb) and design-repository databases for all domains.  This directory has one domain database zip file for each domain.  The files are identified with the six-character code identifying that domain (such as 'amficr' for Claims & Reimbursements).  Each zip file contains one pub db, identified with the four-character identifier for the domain and the two-letter code 'PB' for the pub db (such as FICR_PB.mdb).  If a domain stores any of its static model designs in a design repository, this zip file will also contain the repository database, identified with the four-character identifier for the domain and the two-letter code 'RP' for the repository (such as FICR_RP.mdb).  If a domain stores its model information in the newer XML format saved by the RMIM Designer (in Visio©, then no repository will be present. Please note one or two domains include model information both in a repository (for the domain's older models) and in the RMIM Designer XML (for the newer models).
  • rmimdesignersource – Holds a domain graphics zip file for each domain.  The files are identified with the six-character code identifying that domain (such as amficr for Claims &amp; Reimbursements).  Within the zip file are the DMIM and RMIM Visio© (*.vsd) files, along with the RMIM Designer XML files, for those designs that are not saved in a design repository. 
  • generatorinputfiles – Holds all of the content necessary to run the V3 Generator for the Edition.  The Generator takes a variety of source files and produces Model Interchange Format (MIF) files, design table views, design Excel views and schemas for the Edition.  The structure of the content included here is such that it can be copied or moved into the Generator to reproduce the content for the Edition.  To conserve space on the CD, this content is in a ZIP archive, and is accompanied by a ReadMe file detailing how and where to unzip this. 

Tool Files Subdirectory (Disc 2 only)
(go to top)

The toolfiles directory holds a set of tool installer files (*.msi) and ZIP archives (*.zip) for installing the four primary tools used to develop and publish this Edition.  These are:
  • HL7 Publication Database – The Publication Database (PubDb) is an MS Access© database with a set of HL7-designed forms and Visual Basic© for Applications (VBA) code.  It serves as a vehicle to document the dynamic or interaction model of a technical committee's design.  It allows content developers to create domain content in the form of general domain information and specific topics.  It is distributed here as a Windows Installer© "msi" file hl7_pubdb-.  It is the source of the documentation for the various artifacts such as interactions, trigger events, message types, storyboards, etc. that make up a complete domain specification.  This tool is designed to "enforce" the HL7 development framework such that each specification is internally consistent and consistent with the methodology/
  • HL7 RMIM Designer– The HL7 RMIM Designer is a graphic designer that supports the design of HL7 static information models.  It is distributed here as a Windows Installer© "msi" file hl7_rmimdesigner_visio-.  The current variant of this tool is built upon: customized "templates" in Microsoft Visio© (Versions 2002, 2003 or 2007); a suite of Visual Basic© and VBA software that runs in the application space; and modules (DLL's) from the RoseTree program.  The tool provides an interactive design capability for HL7 RMIMs, and is programmed to use data from the RIM, Vocabulary, and Data Types specifications to enforce the model design rules expressed in the HL7 V3 Methodology.
  • RoseTree – RoseTree is a Visual Basic© program contributed to HL7 by its developer for use in defining HL7 Version 3 specifications.  It is distributed here as a Windows Installer© "msi" file hl7_rosetree-  It serves three functions in support of these specifications:
    1. It provides the primary interface to the HL7 RIM and Vocabulary specifications stored in the design repositories. As such, it supports interactive browsing of this content by the users, as well as programmatic access to the content for the RMIM Designer.
    2. It provides the means for storing, editing and exporting "HMD" designs from the RMIM Designer.  This means of static model design management is gradually being phased out in favor of direct static model expression in XML, but for now, about two-thirds of the static models in this Edition use this storage.
    3. It provides the software for "publishing" the content of the HL7 PubDB in a "domain xml" file that can be processed by the V3 Generator.
  • HL7 V3 Generator– The V3 Generator is an expanding set of tools designed to convert HL7 V3 specifications (from sources such as the RIM, Vocabulary, domain PubDbs and RMIM Designer static model designs) into a coherent and valid set of files in the HL7 Model Interchange Format (MIF), and then to use those files to generate schemas, instance examples, and human-readable views of the specifications.  It is distributed here in ZIP format as hl7_v3Generatortool- and hl7_v3generatorfullsource-.  The Generator itself uses Java tools and an ANT-defined processes to apply a suite of HL7-defined transforms that accomplish these goals.

The install files for these tools are located in the toolfiles directory. This directory also includes a Set-Up and Installation Read Me file for these installers.  Complete documentation on setting up your platform and settings, as well as an overview of the tools is on the HL7 Wiki as Installing and Configuring HL7 Tools (Wiki UserId:'wiki', passWord:'wikiwiki')  It is assumed that the user will already have licensed versions of MS Access© and MS Visio© installed on their systems. Additional releases of these tools, as well as additional information on them are available on the HL7 Project Homebase GForge website.
